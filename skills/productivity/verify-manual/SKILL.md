# 手动验证手册生成器

为刚完成的代码实现生成一份逐步验证手册，存放在 `docs/verify/` 目录下。

## 触发场景

1. 用户主动说"帮我验证"、"验证手册"、"verify"、"手动测一下"
2. `/to-prd` 或 `/to-issues` 完成代码后自动触发

## 前置信息收集

生成手册前，先确认以下信息（如果上下文中已有就直接用，不要重复问）：

1. **Issue/PRD 编号** — 验证手册的文件名依据
2. **改动了什么** — 阅读最近的 git diff 或对话上下文，总结变更范围
3. **应用类型** — Web 前端 / API 后端 / CLI 工具 / 全栈（决定验证方式）

如果无法确定 issue 编号，用 `verify-YYYY-MM-DD.md` 代替。

## 生成流程

### Step 1: 分析变更

运行以下命令了解改了什么：

```bash
git diff --stat HEAD~1
git diff HEAD~1 -- '*.py' '*.ts' '*.tsx' '*.js' '*.jsx'
```

结合对话上下文，总结：
- 新增了哪些功能点
- 修改了哪些现有行为
- 可能影响的关联区域

### Step 2: 生成验证手册

根据分析结果，生成以下结构的验证手册：

```markdown
# 验证手册 · Issue #NNNN — [简短标题]

> **生成时间**: YYYY-MM-DD HH:MM
> **变更范围**: [一句话概括改了什么]
> **应用类型**: [Web 前端 / API 后端 / CLI / 全栈]

---

- **预期**: 服务启动无报错，监听在预期端口
- **截图**: 终端输出截图
- [ ] 通过

**遗留问题**:

```

### Step 3: 写入文件

将生成的手册写入 `docs/verify/issue-NNNN.md`（或 `docs/verify/YYYY-MM-DD.md`）。

如果 `docs/verify/` 目录不存在，先创建：

```bash
mkdir -p docs/verify
```

### Step 4: 输出摘要

在对话中输出：

```
✅ 验证手册已生成：docs/verify/issue-NNNN.md

📋 验证清单概览：
- 🔴 冒烟测试：X 步（先跑这个，5 分钟）
- 🟡 功能点：X 个（逐个验证）
- 🟢 端到端：X 个场景（完整流程）

💡 建议：先跑冒烟测试，通过后逐个验证功能点。
   发现问题直接在手册里打 ❌ 并记录实际结果。
```

## 验证方式参考

根据应用类型，验证步骤的写法不同：

### Web 前端
- 步骤以"打开/点击/填写/提交"为主
- 预期以页面状态/跳转/Toast 提示为主
- 截图指引具体到页面区域

### API 后端
- 步骤以 curl 命令或 Postman 请求为主
- 预期以 HTTP 状态码 + 响应体结构为主
- 截图指引到终端/Postman 响应

### 全栈
- 前端操作 + 后端验证结合
- 既检查 UI 反馈，也检查数据持久化

## 注意事项

- 步骤要**具体到可以盲操**，不能写"测试一下功能"这种模糊描述
- 预期结果要**可判断**，不能写"应该正常"，要写"返回 200，body 包含 `{"id": N}`"
- 冒烟测试步骤控制在 **3 步以内**，确保 5 分钟能跑完
- 功能点验证每个点控制在 **5 步以内**
- 如果变更很小（比如只改了一个字段验证），可以省略端到端场景
